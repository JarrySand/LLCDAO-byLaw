# 3. 提案・投票の進め方（Proposal Flow）

DAOの意思決定は、「提案 → 議論 → 投票 → 執行」のサイクルで進みます。

## 1. 提案（Idea & Proposal）
誰でも提案できます。

1. **アイデア出し**: Discord `#一般` や `#議題提案` で「こんなことやりたい」と投稿する。
2. **ドラフト作成**: 反応が良ければ、フォーラムに正式なスレッドを立てる。
    - **テンプレート**:
        - **タイトル**: 分かりやすく（例: 「マーケティング施策への予算申請」）
        - **概要**: 何をするのか
        - **背景**: なぜやるのか
        - **詳細**: 具体的な手順、必要な予算
        - **期間**: いつやるか
3. **賛同集め**: 「いいね」やコメントで、**5名以上の賛同**を集める（総会規程第4条）。

## 2. 告知と投票設定（Setup）
賛同が集まったら、運営（業務執行社員）が投票の準備をします。ここでのミス（設定間違い）は手戻りになるので慎重に行います。

1. **決議レベルの判定**: 
    - どのレベルで決議すべきか、運営規程やDAO総会別紙を見て判断します。
    - **重要決議**: 定款変更、役員選任、解散、100万円超の支出など。DAOの根幹に関わるため、**期間: 7日以上**が必要です。
    - **通常決議**: ルール変更、予算承認など。通常の運営事項です。（**期間: 3日〜5日**）
2. **Snapshot設定**:
    - Snapshotの管理画面で提案を作成します。
    - **選択肢**: 基本は「賛成 / 反対 / 棄権」ですが、複数案から選ぶ場合はその選択肢を設定します。
    - **開始・終了日時**: 規程で定めた期間（例: 11月1日 0:00 〜 11月8日 23:59）を正確に入力します。
3. **告知（重要！）**:
    - **義務**: 運営規程により、**投票開始の前日まで**に告知しなければなりません。これを忘れると「招集手続きの不備」として投票が無効になるリスクがあります。
    - **場所**: 全員が見ている Discord `#アナウンス` やメーリングリスト。
    - **内容**: 「明日から〇〇についての投票が始まります。期間は〜までです。詳細は（URL）を確認してください。」と明記します。

## 3. 投票期間（Voting）
メンバーはSnapshotで投票します。オフチェーン投票なのでガス代はかかりません。

- **議論の継続**: 
    - 投票期間中も「やっぱりこうした方がいいのでは？」という意見が出ることがあります。Snapshotの設定で、投票期間中の変更（票の入れ直し）を許可しておくと、議論の深化を反映できます。
- **定足数チェック**: 
    - 終了間際（例: 残り24時間）になっても投票率が定足数（例: 20%）に届いていない場合、リマインド（呼びかけ）を行います。
    - 「投票してください！」と呼びかけるのは運営の重要な仕事です。

## 4. 開示と執行（Execution）
投票が終わったら、結果を確定させて実行に移ります。ここもスピード感が大切です。

1. **結果開示**:
    - 終了後**7日以内**に、結果（例: 賛成80%、反対20%で可決）を `#アナウンス` で報告します。
    - **議事録**: 「いつ、どんな議題で、結果はどうだったか」を簡単なメモ（マークダウン形式など）で残し、GitHub等に保存します。これが後で「ちゃんと決めた証拠」になります。
2. **執行**:
    - **可決された場合**: 業務執行社員は、決議内容に従って速やかに実行します（送金、契約締結など）。運営規程第6条により、通常決議なら30日以内、重要決議なら14日以内に着手する義務があります。
    - **否決された場合**: 提案は廃案になります。提案者は、フィードバックを受けて修正し、再提案することも可能です。
    - **拒否権の発動**: 万が一、決議内容が「法令違反」や「DAOに著しい損害を与える」場合、業務執行社員は拒否権を行使できます（定款第19条）。ただし、これは伝家の宝刀であり、乱用すれば信頼を失います。使う場合はしっかりとした説明責任が伴います。
