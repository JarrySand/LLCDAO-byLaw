## 実運用ガイド（Operation）

本ドキュメントは、定款・各規程・別紙の日常運用および変更管理の実務手順、チェックリスト、注意点を示します。公開コメントの対象に含まれるため、透明で再現可能なプロセス記述を心がけます。

注: 本文書は参考資料であり、規程本文ではありません（規範性は規程が優先します）。

### 適用範囲と前提
- 対象ファイル: リポジトリ直下の `01_*.md`（定款）・`02_*.md`（DAO総会規程）・`03_*.md`（運営規程）・`04_*.md`（トークン規程）、および `appendices/` 配下の別紙群。
- 補助資料: `docs/guide/context.md`（設計思想）、`docs/guide/structure.md`（構成/参照/版管理）。
- 公開コメント: `docs/guide/`（context/structure/operation）もレビュー対象に含まれます（`docs/README.md` 参照）。

---

### 1. 変更提案フロー（Issue → Branch → PR → レビュー → 決裁 → リリース）

1) 事前準備（影響範囲と整合性の確認）
- 改訂対象と根拠を明確化（どの条・どの別紙・どの定義か）。
- 整合性チェック: 用語の一貫性、別紙との齟齬、版番号・発効日の整合。
- 参照関係確認: `structure.md` の参照方針に反しないか。
- 変更種別の分類: 重大（定款/投票要）・通常（規程/多者レビュー）・軽微（誤字/体裁/非機能）。

2) Issue 起票（テンプレ＋ラベル）
- タイトル: 「規程名 x.y → x.z（要約）」
- 本文: 背景・目的、変更点サマリ、影響範囲、代替案、リスク、移行手順。
- 添付: 差分プレビュー（可能なら `tools/diff_to_md.py` 出力）、関連資料リンク。
- ラベル例: `major`/`minor`/`chore`、`bylaws`/`assembly`/`operation`/`token`、`appendix`、`breaking-change`。

3) ブランチ戦略
- ベース: `release` ブランチから派生。
- 命名: `feat/operation-...` `fix/token-...` `docs/appendix-...` など。
- コミット原則: 1コミット1論点、原文変更と体裁調整は分離。

4) 編集ガイドライン（本文の書き方）
- 不要な書式変更を避ける（差分最小化、インデント/改行スタイルは現状踏襲）。
- 条・項番号、見出し階層、用語は既存規約に一致させる。
- 版情報・発効日の更新（必要時）。
- 別紙へ移管・追補する場合は、本文からの適切な参照（「別紙〇〇を参照」）に更新。

5) PR 作成
- テンプレ項目: 変更目的、差分ハイライト、整合性チェック結果、影響範囲、移行/周知計画。
- 機械的差分の添付: `tools/diff_to_md.py` による可読差分（任意）。
- レビュー観点: 法的整合、相互参照、運用実装可能性、リスク。

6) レビューと修正
- 最低2名のレビューを推奨（法務/運用/技術など異なる観点）。
- 指摘は論点単位でレス。論点が分岐したらサブIssue化を検討。
- 合意形成後、PR内で結論・最終文言を記録。

7) 決裁（ガバナンス）
- 決裁権限: 規程別に定める（例: 定款改訂は DAO 総会決議）。
- 決裁証跡: 決議リンク/署名情報/投票記録をPRに添付。

8) リリース（反映とタグ付け）
- マージ先: `release`。必要に応じてタグ `vX.Y.Z-docs` を付与。
- アーカイブ: 旧版を `archive/` に保存（フォルダ命名は実施日準拠）。
- 公開: レポジトリ公開範囲での告知。必要に応じて外部媒体（例: Notion）へ反映。

9) 周知と移行
- 変更通知: サマリ、発効日、影響対象、アクション要求を明記。
- 追随タスク: 別紙更新、運用手順書更新、教育・説明会など。

チェックリスト（変更提案）
- [ ] Issue の目的・影響範囲が明確
- [ ] 差分が最小・可読
- [ ] 参照関係と用語整合が保たれている
- [ ] 承認権限/決議手続が確認済
- [ ] リリース・周知手順が用意されている

### 2. DAO総会の開催・記録運用

開催前
- 通知: 規程に定める期限・手段で招集（議題案内、資料、投票方法）。
- 議題確定: 提案の適格性確認、重複/依存関係の整理、所要決議種別の明確化。
- 議決権確定: 権利確定日（記録日）・保有要件（トークン等）・委任の取扱い。
- 資料配布: 提案本文、影響分析、想定Q&A、ドラフト決議文。

当日運用
- 出席/委任確認と定足数確認（検証手続の記録）。
- 議事進行: 議題ごとに説明→質疑→修正動議→採決。
- 投票: 規程に従う（オンチェーン/オフチェーン）。結果と根拠の保存。

終了後
- 議事録: 署名要件・保存形式・公開範囲の順守。修正確定文言を明記。
- 反映: 改訂が可決された場合、上記「変更提案フロー」のリリース手続へ。
- 公開: 公開対象・媒体・期間を規程に従い実施。

チェックリスト（総会）
- [ ] 招集通知の期限・方式が規程どおり
- [ ] 定足数/議決権確認の証跡が保存
- [ ] 投票結果の算出根拠が検証可能
- [ ] 議事録が署名・公開要件を満たす

### 3. 権限・トークン運用（発行・付与・回収・監査）

基本方針
- 最小権限、可監査性、可撤回性。台帳と規程の整合を常に保つ。

実務手順
- 発行/焼却: 承認権限・上限・手続に適合。トランザクションハッシュを記録。
- 付与/回収: 名寄せ（本人確認方法）とウォレット検証。権限移譲/委任の記録。
- パラメータ更新: レート、クォーラム、閾値などは別紙に集約し、変更時は参照箇所と同時更新。

監査
- 定期: 月次/四半期で発行量・ロック・権限付与状況を照合。
- 例外: インシデント時は凍結・回収・権限制限の手続きを優先実行。

チェックリスト（トークン/権限）
- [ ] 取引/更新ごとに証跡（TxID/承認記録）が残る
- [ ] 別紙パラメータと実装値が一致
- [ ] 回収/撤回の実行手段が事前に準備済

### 4. コンプライアンス・リスク対応

運用基準
- 情報公開: 公開コメント対象の範囲と手続きを遵守。
- 利益相反: 提案者/議決者の利害関係の開示と回避措置。
- データ保護: 個人情報・機微情報は公開対象から分離保護。

インシデント対応フロー
1) 検知/報告 → 2) 一時的緩和（凍結・告知） → 3) 原因分析 → 4) 恒久対策 → 5) 再発防止の規程反映。

エスカレーション
- 重大度に応じて承認権限・総会決議・外部公表の要否を判定。

---

### 5. よくある失敗と回避策

- 文字装飾やインデント変更で不要な差分が膨らむ → 体裁変更は別PRに分離。
- 別紙更新の失念 → 変更点に関連するパラメータ/定義の所在をIssue作成時に棚卸。
- 参照関係の断絶 → `structure.md` を参照し、双方向参照や循環参照を作らない。
- 決裁証跡の欠落 → 投票記録/署名/承認ログをPRに必ず添付。

### 6. 定期運用チェックリスト

日常（都度）
- [ ] PR/Issue のレビュー遅延がない
- [ ] 変更の影響コミュニケーションが実施済

月次/四半期
- [ ] トークン/権限の棚卸（実装と別紙の一致）
- [ ] 未解決Issueの棚卸と優先順位更新

イベント駆動（総会/重要改訂）
- [ ] 招集・定足数・議決・議事録・公開の一連が完了
- [ ] 反映と周知がリポジトリ/外部媒体で完了

---

### 7. 補助ツール・テンプレート

- 差分生成: `tools/diff_to_md.py`（可読なMarkdown差分の作成に活用可）。
- レビュー資料: `archive/review/` 配下のフォーマット（比較・チェックリスト等）。
- 外部公開/同期: 必要に応じて外部システムへ同期（例: Notion 連携運用がある場合は該当手順書を参照）。
